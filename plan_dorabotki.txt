ПЛАН ДОРАБОТКИ ПРОЕКТА «САЙТ ДЛЯ ЗАМЕНЫ ПРЕПОДАВАТЕЛЯ»
================================================================================
Дата анализа: 15.05.2026
Версия проекта: 1.0.0 (package.json)
Статус: рабочий прототип с тестовыми данными, часть функций реализована на бэкенде,
         но не доведена до продакшн-уровня.

================================================================================
1. КРАТКОЕ ОПИСАНИЕ ТЕКУЩЕГО СОСТОЯНИЯ
================================================================================

Назначение: веб-система для оформления замены преподавателя на занятии —
поиск свободного коллеги по расписанию и компетенциям, создание заявки,
принятие/отклонение замены, обновление расписания.

Стек:
  - Backend: Node.js, Express (server.js)
  - БД: PostgreSQL (database.sql, init_db.js)
  - Frontend: статические HTML + vanilla JS (public/)
  - Аутентификация: JWT (auth.js), пароли упрощены для демо

Структура репозитория (основные файлы, без node_modules):
  - server.js          — API и раздача статики
  - auth.js            — логин и middleware JWT
  - init_db.js         — пул PostgreSQL, инициализация БД
  - database.sql       — схема, сиды, индексы
  - package.json       — зависимости и скрипты (start, dev, init-db)
  - .env               — параметры подключения к БД (не в git по best practice)
  - public/index.html  — страница входа
  - public/dashboard.html — панель: расписание, заявки, новая заявка
  - public/*.js        — модули UI (auth, dashboard, schedule, requests, new-request, profile)
  - public/style.css   — общие стили
  - README             — только заголовок
  - technical_task.txt — пустой

Реализовано (работает при настроенной БД):
  [+] Вход по email + пароль (демо: teacher123 для преподавателей)
  [+] Просмотр расписания преподавателя на выбранную дату
  [+] Навигация по датам (жёстко заданный диапазон в schedule.js)
  [+] Список «Мои заявки» (свои + где выбрали для замены; админ видит все)
  [+] Отмена своей заявки (pending)
  [+] Принятие/отклонение заявки замещающим преподавателем
  [+] Форма новой заявки: поиск преподавателей по коэффициенту, выбор, отправка
  [+] API подбора преподавателей (/api/teachers/available)
  [+] Обновление timetable при подтверждении замены
  [+] Health-check (/api/health)

Частично / с ограничениями:
  [~] Модуль «Новая заявка» — JS реализован (new-request.js), в dashboard.html
      осталась заглушка «в разработке» (устаревший текст)
  [~] Профиль (profile.js) — UI есть, но не подключён к dashboard; API PUT /profile нет
  [~] Администратор — роль в БД и фильтр в API, отдельного UI нет
  [~] ТЗ (technical_task.txt) — не заполнено

Не реализовано / отсутствует:
  [-] Недельный вид расписания (CSS есть, логики нет)
  [-] История замен в UI (/api/schedule/replacements есть, фронт не использует)
  [-] Уведомления (email/push)
  [-] Тесты (unit/e2e)
  [-] Документация по развёртыванию
  [-] .gitignore
  [-] Миграции БД (только пересоздание через DROP в database.sql)

2. ВЫЯВЛЕННЫЕ ПРОБЛЕМЫ И РИСКИ (ПРИОРИТЕТ ИСПРАВЛЕНИЯ)
================================================================================

Критичные (блокируют корректную работу или безопасность):
  1. auth.js — метод getCurrentUser: обращение к несуществующей переменной user
     вместо result.rows[0]; метод объявлен как экземплярный, а server.js вызывает
     Auth.getCurrentUser() как статический → /api/profile падает.
  2. Пароль администратора: на index.html указан admin123, в auth.js принимается
     только teacher123 → вход админа с карточки не работает.
  3. JWT_SECRET захардкожен в auth.js; password_hash в БД не используется,
     bcrypt подключён в package.json, но не применяется.
  4. init_db.js логирует пароль БД в консоль — утечка при деплое/логах.
  5. Публичный /api/test-schedule без авторизации — лишняя поверхность атаки.

Высокие (логика бизнеса / целостность данных):
  6. При принятии замены поле timetable.teacher перезаписывается конкатенацией
     «Имя → Заменяющий» — ломает связь с teachers и последующие запросы.
  7. Нет внешних ключей: teacher_name, replacing_teacher — TEXT без FK на teachers.
  8. lesson_type в подборе преподавателей всегда 'seminar' — неточный рейтинг.
  9. schedule.js не вызывает /api/schedule/available-dates — даты захардкожены
     (15–26.12.2025), расхождение с реальным расписанием в БД.
  10. Нет API и UI для действий администратора (модерация, назначение замены,
      смена статуса вручную), хотя роль admin в запросах учтена.

Средние (UX, сопровождение):
  11. Класс .card используется в requests.js и new-request.js, в style.css не описан.
  12. Дублирование CSS: большие блоки в dashboard.html и style.css.
  13. profile.js не подключён; GET /api/profile сломан; PUT /api/profile отсутствует.
  14. viewRequestDetails / showRequestModal в requests.js не завершены.
  15. helmet и morgan в зависимостях, в server.js не подключены.
  16. Избыточное логирование (пароли, токены, тела запросов) в production-режиме.
  17. Эвристика «gender_bonus» по окончанию фамилии — спорная и может быть
      неприемлема для продакшна; лучше убрать или вынести в настраиваемые правила.
  18. CORS жёстко на http://localhost:3000 — не готово к другому хосту/порту.

Низкие (гигиена репозитория):
  19. README и technical_task.txt не описывают требования.
  20. Нет .gitignore (риск коммита node_modules, .env).
  21. npm test — заглушка.
  22. database.sql при каждом init-db уничтожает все данные (DROP CASCADE).

--------------------------------------------------------------------------------
ЭТАП 1. Стабилизация и безопасность (3–5 дней)
--------------------------------------------------------------------------------
Цель: убрать критические баги, подготовить к тестовой эксплуатации.

Задачи:
  1.1. Исправить Auth.getCurrentUser (static, корректное чтение из БД + join teachers).
  1.2. Единая политика паролей:
       - хеширование bcrypt при создании пользователей;
       - login сравнивает password с password_hash;
       - отдельные тестовые пароли в database.sql или seed-скрипте;
       - синхронизировать подсказки на index.html с реальной логикой.
  1.3. JWT_SECRET и срок жизни токена — только из process.env (.env.example).
  1.4. Удалить или защитить /api/test-schedule (только dev + флаг ENABLE_TEST_ROUTES).
  1.5. Подключить helmet, ограничить cors через env (ALLOWED_ORIGINS).
  1.6. Убрать логирование паролей и полных токенов; оставить структурированные логи
       (morgan в dev, без чувствительных данных).
  1.7. Валидация входных данных (express-validator или zod): даты, classes 1–8,
       обязательные поля заявки, enum status/action.

Результат: безопасный вход, рабочий /api/profile (GET), предсказуемая конфигурация.

--------------------------------------------------------------------------------
ЭТАП 2. Целостность данных и бизнес-логика замены (5–7 дней)
--------------------------------------------------------------------------------
Цель: корректная модель замены в БД и API.

Задачи:
  2.1. Рефакторинг схемы timetable при замене:
       - не портить поле teacher;
       - хранить original_teacher_id / replacement_teacher_id или отдельную таблицу
         lesson_replacements (рекомендуется для аудита);
       - флаг is_replacement и связь с replacement_requests.id.
  2.2. Добавить FK: replacement_requests.teacher_name → teachers(name),
       replacing_teacher → teachers(name); CHECK по status
       (pending, confirmed, rejected, cancelled).
  2.3. Учитывать lesson_type из timetable или отдельного справочника занятий
       при запросе /api/teachers/available.
  2.4. Пересмотреть формулу коэффициента: база из teacher_competencies + опыт;
       убрать или параметризовать gender_bonus; документировать в README/ТЗ.
  2.5. Проверки при создании заявки: занятие существует у инициатора на дату/пару;
       нельзя два pending на одну пару; замещающий не занят (уже частично есть).
  2.6. Транзакции и идемпотентность respond/cancel (защита от двойного клика).
  2.7. Разделить init-db: schema.sql (CREATE) + seeds.sql (INSERT) + migrations/
       (node-pg-migrate или knex) вместо DROP на каждый запуск.

Результат: замена не ломает расписание; история замен восстанавливаема из БД.

--------------------------------------------------------------------------------
ЭТАП 3. Frontend — завершение пользовательских сценариев (5–8 дней)

Задачи:

       кнопки prev/next по реальному списку дат.
  3.2. dashboard.html: убрать заглушку «модуль в разработке» в секции новой заявки;
       при первом открытии раздела вызывать loadData() у NewRequestModule.
  3.3. Добавить стили .card, .btn-primary в style.css; вынести inline-стили из JS
       в CSS по возможности.
  3.4. «Мои заявки»: модальное окно деталей (viewRequestDetails); фильтры по статусу

  3.5. Блок «История замен» на основе GET /api/schedule/replacements (вкладка или

  3.6. Опционально: недельный вид расписания (использовать готовые CSS-классы).
  3.7. Профиль: секция в dashboard + profile.js; API PUT /api/profile (телефон в teachers).
  3.8. Обработка ошибок сети, истечения JWT (редирект на /, сообщение «сессия истекла»).
  3.9. Адаптивность: проверить мобильную вёрстку навигации и карточек заявок.

Результат: полный цикл «расписание → заявка → ответ замещающего» без ручных обходов.

--------------------------------------------------------------------------------
ЭТАП 4. Панель администратора (4–6 дней)
--------------------------------------------------------------------------------
Цель: управление заявками и справочниками для роли admin.

Задачи:
  4.1. UI для admin: список всех заявок с фильтрами (дата, статус, преподаватель).
  4.2. API (при необходимости):

       - назначение replacing_teacher админом;
       - отмена/принудительное закрытие.
  4.3. CRUD преподавателей и компетенций (минимум: просмотр, редактирование coefficient).
  4.4. Импорт/обновление расписания (CSV или форма пакетной загрузки) — по ТЗ.
  4.5. Скрывать кнопки «замена» для admin без teacher_name или показывать режим
       «только просмотр».

Результат: админ может сопровождать процесс без правки БД вручную.

Задачи:

  5.3. Опционально: экспорт отчёта замен за период (CSV/PDF).
  5.4. Опционально: интеграция с корпоративным LDAP/OAuth вуза.

Результат: снижение зависимости от ручного контроля личного кабинета.

--------------------------------------------------------------------------------
ЭТАП 6. Качество, деплой, сопровождение (ongoing, 3–5 дней базово)
--------------------------------------------------------------------------------
Задачи:
  6.1. Тесты: Jest/Supertest для API (login, create request, respond, schedule);
       Playwright/Cypress — сценарий входа и создания заявки.
  6.2. CI: lint (eslint), тесты на push; не коммитить node_modules.
  6.3. Docker Compose: app + postgres для локального и серверного деплоя.
  6.4. Переменные окружения: .env.example без секретов.
  6.5. Резервное копирование БД; мониторинг /api/health.
  6.6. Версионирование API (префикс /api/v1) при появлении внешних клиентов.

Результат: воспроизводимый деплой и регрессионная защита.

6. КРИТЕРИИ ГОТОВНОСТИ MVP
================================================================================

  [ ] Преподаватель входит по индивидуальному паролю (не общий teacher123).
  [ ] Видит только свои даты и пары из БД.
  [ ] Создаёт заявку с выбором замещающего из ранжированного списка.
  [ ] Замещающий принимает/отклоняет; расписание обоих корректно обновляется.
  [ ] Инициатор отменяет pending-заявку.
  [ ] Админ видит все заявки и может вмешаться при отказе.
  [ ] Нет критических уязвимостей (секреты в env, тестовые роуты отключены).
  [ ] README позволяет развернуть проект с нуля за < 30 минут.
